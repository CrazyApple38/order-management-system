# 受注管理システム 要件定義書

## 1. プロジェクト概要

### 1.1 システム名
受注管理システム (Order Management System)

### 1.2 目的
現在Excel（VBA）で運用している受注簿・業務管理計画書をWebシステム化し、
データの一元管理、検索性向上、複数取引先の統合管理を実現する。

### 1.3 開発方針

- **Phase 1（先行開発）**: React + Node.js によるWebアプリケーションとして業務管理計画書UIを構築
- **Phase 2（後続開発）**: Excel VBA によるUIを構築（既存の受注簿・計画書との連携）
- Webアプリケーション（React）を正（プライマリ）とし、Excel UIはサブシステムとして位置づける

### 1.4 技術スタック

| 項目 | 技術 | 備考 |
| ---- | ---- | ---- |
| フロントエンド | Next.js (React) | **先行開発（プライマリUI）** |
| バックエンド | Supabase (PostgreSQL + Auth + Storage) | |
| 認証 | Supabase Auth (メール・パスワード) | |
| 外部連携 | Googleカレンダー API | |
| Excel連携 | VBA + Supabase REST API | **後続開発（セカンダリUI）** |
| ホスティング | 未定 (Vercel推奨) | |

## 2. システム要件

### 2.1 利用者
- **対象**: 社内管理者のみ
- **同時利用者数**: 少人数（1〜数名）
- **権限管理**: 初期は単一権限、将来的に役割別権限も検討

### 2.2 動作環境
- **PC**: Chrome, Edge, Firefox（最新版）
- **モバイル**: レスポンシブ対応（iOS Safari, Android Chrome）

### 2.3 既存システムとの関係

- **React Webアプリ（プライマリ）**: 業務管理計画書の主要UIをReact + Node.jsで先行開発
- **並行運用**: React Webアプリ完成後も、当面はExcelシステムと並行して運用
- **データ連携**: Excel形式でのインポート・エクスポート機能を提供（差分処理付き）
- **Excel UI（後続開発）**: React版の完成後に、Excel VBA版のUIを構築しDBと連携

## 3. 機能要件

### 3.1 受注簿管理

#### 3.1.1 データ項目
| カテゴリ | 項目例 |
|----------|--------|
| 契約先情報 | 契約先名、現場名、住所、連絡先 |
| 人員配置 | 配置人数、勤務時間、シフトパターン |
| 金額情報 | 単価、請求額、支払条件 |
| ステータス | 受注状況、変更履歴 |

#### 3.1.2 信頼度（受注確度）

受注データの確度を3段階で管理し、合計表示に反映する。

| 信頼度 | 説明 | グリッド表示 |
|--------|------|-------------|
| 確定 | 確定した受注 | 通常表示（不透明度100%） |
| 予定（信頼:高） | 高確率の予定 | 文字色を薄く表示（不透明度50%） |
| 予定（信頼:低） | 低確率の予定 | 文字色をさらに薄く表示（不透明度25%） |

**自動確定ルール**:

| ルール | 条件 | 動作 |
|--------|------|------|
| 過去日付ルール | 受注対象日が昨日以前 | 自動的に「確定」に変更 |
| 配置確定ルール | 業務管理計画書で社員が配置された | 自動的に「確定」に変更 |

**合計表示（最小～最大）**:

行合計・日別合計・総合計を「最小値～最大値」形式で表示する。

| 値 | 算出方法 |
|----|----------|
| 最小値 | 信頼度が「確定」のデータのみ合計 |
| 最大値 | 全データ（予定含む）の合計 |

※ 最小値と最大値が同じ場合は単一の数値で表示

#### 3.1.3 行情報編集

受注簿グリッドの固定列（会社・区分・昼夜・契約先名・業務名）をクリックすると行情報編集モーダルが開く。

| 項目 | 操作方法 |
|------|----------|
| 会社 | チップ選択（単一選択） |
| 区分 | チップ選択（単一選択）＋ 新規区分追加ボタン |
| 昼夜 | チップ選択（単一選択） |
| 契約先名 | テキスト入力 |
| 業務名 | テキスト入力（空欄で個別業務） |

※ 区分の新規追加時は、バッジ定義にも自動的に同期される

#### 3.1.4 バッジ（区分子バッジ）

セル編集モーダルで、区分に紐づく子バッジをチップ選択できる。
区分名が親バッジ（読み取り専用）、その下に子バッジ候補が表示される。

**構造**:

| 要素 | 説明 |
|------|------|
| 親バッジ | 区分名をそのまま表示（読み取り専用） |
| 子バッジ | 区分ごとに定義されたバッジ候補（複数選択可） |

**初期データ（子バッジ）**:

| 区分 | 子バッジ |
|------|----------|
| 高速 | 車線規制、路肩規制、ブース規制、保安員 |
| 交通 | 片側交互、通行止め |
| 施設 | （なし、追加可能） |
| イベント | （なし、追加可能） |
| 応援交通 | （なし、追加可能） |

**操作**:

| 操作 | 説明 |
|------|------|
| 子バッジ選択 | チップをクリックしてON/OFF切替（複数選択可） |
| 子バッジ追加 | 「+ バッジ追加」ボタンからpromptで新規バッジ名を入力 |
| グリッド表示 | セル内に選択済みバッジ名を「・」区切りで表示 |

※ 親バッジ（区分）は行情報編集モーダルの区分変更に連動して自動決定される
※ 区分変更時、子バッジ選択はリセットされる
※ DB上では `daily_site_orders.badge_child_ids UUID[]` に選択済み子バッジIDを保存

#### 3.1.6 祝日ハイライト

受注簿グリッドの日付列で土日・祝日に背景色を適用する。ツールバーのトグルボタンでON/OFF可能。

| 種別 | 背景色 |
|------|--------|
| 土曜日 | 薄い青系 |
| 日曜日 | 薄い赤系 |
| 祝日 | 薄い金色系 |

※ 祝日データは `holidays` テーブルで管理
※ ヘッダー行はデータ行より濃い色で表示

#### 3.1.7 夜勤行の表示

受注簿・業務管理計画書で夜勤データを視覚的に区別する。

| 画面 | 表示方法 |
|------|----------|
| 受注簿（月間グリッド） | セル内文字色を赤系（#B5524D）に変更 |
| 業務管理計画書（日次） | 行背景を青系（#2b6cb0）+ 白文字 |

※ 2画面で異なるスタイルを使用する理由: 受注簿はセル単位で密度が高いため文字色で区別、業務管理計画書は行単位で幅があるため背景色で区別

#### 3.1.8 区分デフォルトラベル

セル編集モーダルの「業務詳細」で、区分に応じたデフォルトのラベル名を自動設定する。

| 区分 | デフォルトラベル |
|------|----------------|
| 高速 | 工事名、作業内容 |
| 交通 | 作業内容 |
| 施設 | 業務内容 |

※ ラベルはデフォルト値として使用。ユーザーが自由に変更可能
※ DB上は `site_sub_items.label` で管理（マスタレベル）、`daily_order_sub_items.label` で日毎にカスタマイズ可能

#### 3.8.3 差分確認・反映

受注簿（Excel）とシステム（DB）間のデータ同期時に、差分を分類・表示し、ユーザーが1件ずつ確認・選択して反映する。

| 差分タイプ | 説明 | 操作 |
|------------|------|------|
| あいまい一致 | 契約先名の表記揺れ等で完全一致しないが同一と推定されるレコード | どちらの表記を採用するか選択 |
| 値差分 | 同一レコードだが人数・時間等の値が異なる | 項目ごとに受注簿側/システム側を選択、または手動編集 |
| 受注簿のみ（新規） | 受注簿に存在しシステムに存在しないレコード | 取込対象とするか選択 |
| システムのみ | システムに存在し受注簿に存在しないレコード | 削除対象とするか選択 |

**差分確認UI仕様:**
- 左サイドバー: 差分一覧（タイプ別バッジ色付き）、確認済みステータスアイコン
- 右詳細パネル: 選択中の差分の詳細比較テーブル
- 項目ごとにラジオボタンで受注簿側/システム側を選択可能
- 「編集」ボタンで値を手動編集可能
- 「すべて受注簿側」「すべてシステム側」の一括選択
- 進捗表示（確認済み件数 / 全件数）
- 未確認項目がある場合、反映前に警告表示

- [ ] 差分タイプ分類（あいまい一致/値差分/受注簿のみ/システムのみ）

### 3.12 履歴管理

#### 3.12.1 受注変更履歴・ロールバック

**業務背景**: 営業担当者が契約先から受注情報（何日にどの現場で何名必要か）を受け、システムに入力する。
営業担当者が端末を使えない場合は配置担当者に連絡して代理入力する。
別の営業担当者が同じ契約先から誤った伝言を受けて上書きしたり、聞き間違いによる誤キャンセルが発生する。
これらの誤操作から復旧するため、変更履歴の記録とロールバック機能が必要。

| 項目 | 内容 |
|------|------|
| 追跡対象 | 日毎受注（daily_site_orders）の全カラム変更 |
| 記録内容 | 変更カラム名、変更前の値、変更後の値 |
| 変更理由 | 任意入力（例：「契約先からの連絡で人数変更」「キャンセル連絡あり」） |
| ロールバック | 任意の過去バージョンに戻せる |
| ロールバック権限 | 営業担当者・配置担当者・管理者 |
| 競合対策 | 楽観的ロック（バージョン番号照合） + スマートマージ |

**受注ステータス管理**:

| ステータス | 説明                 |
|------------|----------------------|
| active     | 有効（通常状態）     |
| cancelled  | その日のキャンセル   |

**競合解決（スマートマージ）**:

同時編集でバージョン不一致を検知した場合、変更項目の重複有無に応じて解決方法を提示する。

| ケース       | 条件                         | 解決方法                                       |
|--------------|------------------------------|------------------------------------------------|
| A: 非重複    | 自分と他者の変更項目が異なる | 両方の変更をマージして保存（自動マージ候補）   |
| B: 重複あり  | 同じ項目を双方が変更         | 項目ごとにどちらの値を採用するか選択ダイアログ |

- [ ] 受注情報変更時の自動履歴記録（変更カラム・変更前後の値・変更者・日時）
- [ ] 変更理由の入力フォーム（任意）
- [ ] 変更履歴一覧の表示（受注ごと）
- [ ] 任意の過去バージョンへのロールバック機能
- [ ] ロールバック時の確認ダイアログ（差分プレビュー表示）
- [ ] 楽観的ロックによる同時編集の競合検知
- [ ] 競合時のスマートマージ（非重複項目の自動統合提案）
- [ ] 競合時の手動選択ダイアログ（重複項目の値選択）
- [ ] 受注ステータス管理（active/cancelled）

#### 3.12.2 汎用変更履歴

- [ ] 変更履歴の記録（全テーブル対象の汎用ログ）

### 3.13 データベース構造（テーブル相関関係）

本システムのデータベースは、以下の主要なデータグループで構成される。

#### 3.13.1 取引先・現場系

**取引先（companies）** は複数の **現場（sites）** を保有する。各現場は **区分（site_categories）** によって「施設」「イベント」「交通」「高速」などに分類される。

現場は最大3階層の階層構造を持つことができ、**現場子項目（site_sub_items）** で「工事名」「作業内容」などを管理する。例えば「24-1234（PJ番号）→ 〇〇橋補修工事 → 点検作業」のように連結表示される。

#### 3.13.2 社員・資格系

**社員（employees）** は複数の **資格・スキル（employee_licenses）** を保有できる。資格は **資格種別マスタ（license_types）** で「検定」「運転免許」「スキル」の3カテゴリに分類される。

また、人間関係（相性）により同じ現場に配置すべきでない社員の組み合わせを **社員配置制約（employee_conflicts）** で管理する。この制約は配置をブロックするものではなく、警告アイコンで注意喚起を行う。

#### 3.13.3 受注・配置系（中核）

**日毎受注（daily_site_orders）** は営業担当者が契約先から受けた受注情報を管理する。「どの現場に」「いつ」「何名必要か」「時間帯」を記録する。1つの現場に対して昼・夜の2シフトが設定可能。楽観的ロック（version）による同時編集の競合検知に対応。

**日毎配置（daily_assignments）** は配置担当者が管理する配置計画。日毎受注に1:1で紐づき、現場地図URL（map_url）と配置車両（vehicle_ids）を保持する。

**配置詳細（assignment_details）** は日毎配置に紐づき、「誰を」「どの移動手段で」配置するかの詳細を管理する。1つの日毎配置に対して複数の社員を配置できる。

#### 3.13.4 送迎関係

**送迎関係（transport_relations）** は「誰が誰を迎えに行くか」を記録する。運転者と同乗者の関係を日付ごとに管理し、迎え場所も指定できる。

#### 3.13.5 車両・メンテナンス系

**車両（vehicles）** は社有車の情報を管理する。車両は日毎配置（daily_assignments.vehicle_ids）に紐付けられ、1つの現場に複数の車両を配置することも可能。

**メンテナンス予定（maintenance_schedules）** は車検・オイル交換・タイヤ交換などの予定を管理する。Googleカレンダーとの連携も可能。

#### 3.13.6 ペナルティ系

**ペナルティコード（penalty_codes）** は「病欠」「遅刻」などの種別を定義する。**ペナルティ記録（penalty_records）** は社員ごとの発生履歴を記録し、関連する配置との紐付けも可能。

#### 3.13.7 受注変更履歴系

**受注変更履歴（order_change_history）** は日毎受注（daily_site_orders）の全変更を記録する。変更されたカラム名・変更前の値・変更後の値を保持し、任意の過去バージョンへのロールバックを可能にする。楽観的ロック（daily_site_orders.version）により同時編集の競合を検知する。

#### 3.13.8 ログ系

**変更履歴（change_history）** は全テーブルのデータ変更（登録・更新・削除）を自動記録する。変更前後の値をJSON形式で保持。

**通知ログ（notification_logs）** はLINE通知などの送信履歴を記録する。

#### 3.13.9 連続勤務（自動検知）

連続勤務はテーブルで管理せず、配置データから自動的に検知する。「昼→夜」「夜→昼」「昼→夜→昼」「夜→昼→夜」のパターンを検知し、アイコン・バッジと背景色で視覚的に表示する。
