# 開発手順書

受注管理システム（Order Management System）の開発プロセスを定義する。

---

## 全体フロー

```text
Phase 1: 基礎設計
  要件定義・DB設計の初版作成
    ↓
Phase 2: モックアップ開発サイクル（反復）
  各画面のモックアップ作成 → フィードバック → 要件定義・DB設計更新 → 整合性修正
    ↓  ※ユーザーが「全モックアップ完了」を宣言
Phase 3: 仕様書・設計書作成
  3a.機能仕様書 → 3b.画面設計書 → 3c.認証・権限設計書 → 3d.API設計書 → 3e.インフラ設計書
    ↓  ※ユーザーが「仕様書作成完了」を宣言
Phase 4: 本番環境構築
  4a.環境構築 → 4b.認証・基盤 → 4c.画面実装+単体テスト → 4d.結合・E2Eテスト → 4e.デプロイ
```

---

## Phase 1: 基礎設計

要件定義とデータベース設計の初版を作成する。

### 成果物
| ファイル | 内容 |
|---------|------|
| `docs/01_要件定義.md` | システム要件定義 |
| `docs/02_既存システム分析.md` | Excel VBA既存システムの分析 |
| `docs/03_データベース設計.md` | PostgreSQL / Supabase DB設計 |
| `docs/04_差分処理要件定義.md` | 差分処理の要件定義 |

### 完了条件
- 要件定義・DB設計の初版が作成されていること
- ※Phase 2で各モックアップのフィードバックにより継続的に更新される

---

## Phase 2: モックアップ開発サイクル

各画面のモックアップを1つずつ作成し、アイデアを盛り込みながら反復的に改善する。

### モックアップ一覧（3〜5画面予定）

| ID | 画面名 | 状態 |
|----|--------|------|
| A | 業務管理計画書 | 作成中 |
| B | 受注簿 | 作成中 |
| C | Quick Access（受注クイック入力） | — |
| D | 経理画面（グループ間請求確認） | — |
| E | 休日申請管理 | — |

### 各モックアップの開発手順

```
1. モックアップ作成（仮データ使用）
   ↓
2. アイデア出し・機能追加
   ↓
3. ユーザーが「このモックアップは完了」と宣言
   ↓
4. フィードバック反映（自動実行）
   → 要件定義（01）を更新
   → DB設計（03）を更新
   → 必要に応じて他のドキュメントも更新
   ↓
5. 既存モックアップとの整合性確認（事前報告 → 承認後に反映）
   → 後のモックアップで生まれた変更が既存モックアップに影響する場合、
     変更内容を報告してからユーザー承認を得て反映する
```

### ゲート条件
- **次のモックアップへの進行**: ユーザーが現在のモックアップの完了を宣言すること
- **Phase 3への進行**: ユーザーが「全モックアップ完了」を明示的に宣言すること

### 注意事項
- モックアップは Vanilla HTML/CSS/JS で作成（本番とは別）
- コード内の `【モックアップ専用】` `【検証用】` マークは本番環境には含めない
- 仮データを使用し、実際のDB接続は行わない

---

## Phase 3: 仕様書・設計書作成

完成したモックアップ・要件定義・DB設計をもとに、本番開発に必要なドキュメントを作成する。
依存関係があるため、以下の順序で作成する。

### 作成順序と成果物

```
3a. 機能仕様書
 ↓
3b. 画面設計書
 ↓
3c. 認証・権限設計書
 ↓
3d. API設計書
 ↓
3e. インフラ設計書
```

| Step | ドキュメント | 内容 | 依存先 |
| ---- | ----------- | ---- | ------ |
| 3a | 機能仕様書 | 各画面の操作フロー・ビジネスロジック・データ処理ルール | モックアップ・要件定義 |
| 3b | 画面設計書 | レイアウト詳細・コンポーネント一覧・状態遷移図 | 3a |
| 3c | 認証・権限設計書 | Supabase Auth設定・ロール定義・アクセス制御マトリクス | 3a |
| 3d | API設計書 | エンドポイント一覧・リクエスト/レスポンス定義・エラーハンドリング | 3b, 3c |
| 3e | インフラ設計書 | Supabaseプロジェクト設定・Storage構成・RLSポリシー定義 | 3c, 3d |

### ゲート条件
- **Phase 4への進行**: ユーザーが「仕様書作成完了」を明示的に宣言すること

---

## Phase 4: 本番環境構築

完成した仕様書・設計書・モックアップをもとに、本番環境を段階的に構築する。

### 技術スタック
- **Frontend**: Next.js 14+ (App Router), TypeScript, Tailwind CSS
- **Backend**: Supabase (PostgreSQL, Auth, Storage)

### 実装ステップ

```text
4a. 環境セットアップ
 ↓
4b. 認証・基盤機能実装
 ↓
4c. 画面単位の実装（各画面ごとに単体テスト）
 ↓
4d. 結合テスト・E2Eテスト・修正
 ↓
4e. デプロイ・本番稼働
```

| Step | 工程 | 内容 | テスト |
| ---- | ---- | ---- | ------ |
| 4a | 環境セットアップ | Supabaseプロジェクト作成・Next.jsプロジェクト初期化・DB マイグレーション・開発環境構築 | 接続確認 |
| 4b | 認証・基盤機能実装 | Supabase Auth設定・ロール実装・共通コンポーネント・レイアウト | 認証フローの単体テスト |
| 4c | 画面単位の実装 | モックアップごとに画面を実装（A→B→C...の順） | 各画面の単体テスト |
| 4d | 結合テスト・E2Eテスト | 画面間の連携テスト・エンドツーエンドテスト・バグ修正 | 結合テスト・E2Eテスト |
| 4e | デプロイ・本番稼働 | 本番環境へのデプロイ・動作確認・運用開始 | 本番環境での最終確認 |

### データ移行（未定）

- 既存Excel VBAデータの移行要否はPhase 4開始時に判断する
- 必要な場合、4aと4bの間に移行ステップを追加する

---

## ルール

### フェーズ進行ルール
1. **ユーザー宣言なしにフェーズを進めない**
   - 「モックアップ完了」「仕様書作成に進む」等の明示的な宣言が必要
2. **フィードバック反映は自動で進めてよい**
   - モックアップ完了宣言後の要件定義・DB設計への反映は承認不要
3. **モックアップ間の整合性修正は事前報告**
   - 後のモックアップの変更を既存モックアップに反映する場合、変更内容を報告し承認を得てから実施

### 名付け・用語ルール
- 新しい用語や名前を導入する際、既存の用語との整合性を確認する
- 整合性が取れない場合や判断が必要な場合は、ユーザーにステップバイステップで確認する
- 自己判断で用語を統一・変更しない

### 改善提案ルール
- より良いアプローチや改善点がある場合、ステップバイステップで提案・確認する
- ユーザーの理解度が90%を超えるまで確認を続ける
