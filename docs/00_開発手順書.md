# 開発手順書

受注管理システム（Order Management System）の開発プロセスを定義する。

---

## 全体フロー

```
Phase 1: 基礎設計
    ↓
Phase 2: モックアップ開発サイクル（反復）
    ↓  ※ユーザーが「全モックアップ完了」を宣言するまで繰り返し
Phase 3: 仕様書・設計書作成
    ↓  ※ユーザーが「仕様書作成完了」を宣言するまで
Phase 4: 本番環境構築
```

---

## Phase 1: 基礎設計

要件定義とデータベース設計の初版を作成する。

### 成果物
| ファイル | 内容 |
|---------|------|
| `docs/01_要件定義.md` | システム要件定義 |
| `docs/02_既存システム分析.md` | Excel VBA既存システムの分析 |
| `docs/03_データベース設計.md` | PostgreSQL / Supabase DB設計 |
| `docs/04_差分処理要件定義.md` | 差分処理の要件定義 |

### 完了条件
- 要件定義・DB設計の初版が作成されていること
- ※Phase 2で各モックアップのフィードバックにより継続的に更新される

---

## Phase 2: モックアップ開発サイクル

各画面のモックアップを1つずつ作成し、アイデアを盛り込みながら反復的に改善する。

### モックアップ一覧（3〜5画面予定）

| ID | 画面名 | 状態 |
|----|--------|------|
| A | 業務管理計画書 | 作成中 |
| B | 受注簿 | 作成中 |
| C | （未定） | — |
| D | （未定） | — |
| E | （未定） | — |

### 各モックアップの開発手順

```
1. モックアップ作成（仮データ使用）
   ↓
2. アイデア出し・機能追加
   ↓
3. ユーザーが「このモックアップは完了」と宣言
   ↓
4. フィードバック反映（自動実行）
   → 要件定義（01）を更新
   → DB設計（03）を更新
   → 必要に応じて他のドキュメントも更新
   ↓
5. 既存モックアップとの整合性確認（事前報告 → 承認後に反映）
   → 後のモックアップで生まれた変更が既存モックアップに影響する場合、
     変更内容を報告してからユーザー承認を得て反映する
```

### ゲート条件
- **次のモックアップへの進行**: ユーザーが現在のモックアップの完了を宣言すること
- **Phase 3への進行**: ユーザーが「全モックアップ完了」を明示的に宣言すること

### 注意事項
- モックアップは Vanilla HTML/CSS/JS で作成（本番とは別）
- コード内の `【モックアップ専用】` `【検証用】` マークは本番環境には含めない
- 仮データを使用し、実際のDB接続は行わない

---

## Phase 3: 仕様書・設計書作成

完成したモックアップ・要件定義・DB設計をもとに、本番開発に必要なドキュメントを作成する。

### 成果物（予定）
| ドキュメント | 内容 |
|-------------|------|
| 仕様書 | システム全体の機能仕様 |
| 画面設計書 | 各画面のレイアウト・操作仕様 |
| API設計書 | エンドポイント・リクエスト/レスポンス定義 |
| その他 | 必要に応じて追加 |

### ゲート条件
- **Phase 4への進行**: ユーザーが「仕様書作成完了」を明示的に宣言すること

---

## Phase 4: 本番環境構築

完成した仕様書・設計書・モックアップをもとに、本番環境を構築する。

### 技術スタック
- **Frontend**: Next.js 14+ (App Router), TypeScript, Tailwind CSS
- **Backend**: Supabase (PostgreSQL, Auth, Storage)

---

## ルール

### フェーズ進行ルール
1. **ユーザー宣言なしにフェーズを進めない**
   - 「モックアップ完了」「仕様書作成に進む」等の明示的な宣言が必要
2. **フィードバック反映は自動で進めてよい**
   - モックアップ完了宣言後の要件定義・DB設計への反映は承認不要
3. **モックアップ間の整合性修正は事前報告**
   - 後のモックアップの変更を既存モックアップに反映する場合、変更内容を報告し承認を得てから実施

### 改善提案ルール
- より良いアプローチや改善点がある場合、ステップバイステップで提案・確認する
- ユーザーの理解度が90%を超えるまで確認を続ける
