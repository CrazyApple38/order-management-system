# 差分処理 要件定義書

## 1. 概要

### 1.1 目的

受注簿（Excel）と本システム（Supabase DB）間でデータを同期する際、差分が存在する場合にそのまま上書きせず、ユーザーが項目ごとに反映内容を選択・編集できる差分処理機能を提供する。

> **実装方針**: 差分処理UIはReact Webアプリ（先行開発）のモーダルコンポーネントとして実装する。後続開発のExcel VBA版でも同等の差分処理UIを構築する。

### 1.2 対象操作

| 操作 | 方向 | 説明 | UI |
| --- | --- | --- | --- |
| 受注簿データ取得 | Excel → DB | Excelファイルをアップロードし、DBと比較して差分を処理 | React Webアプリ |
| 受注簿へ転記 | DB → Excel | DBのデータをExcelに書き出す際、Excelの既存データと比較して差分を処理 | React Webアプリ |

※ 両操作とも同一の差分UIを使用する（方向ラベルのみ異なる）
※ React Webアプリではブラウザ上でExcelファイルの読み込み・書き出しを行う（サーバーサイドでのExcelパース）

### 1.3 前提条件

- データソース: Excel ↔ Supabase DB の双方向同期
- 両方で編集される（Excelは営業側、React Webアプリ/DBは配置担当側）
- 取得タイミング: 初回取り込み + 随時（受注内容変更時）
- 差分処理の操作はReact Webアプリ上で行う（先行開発）

## 2. 差分が発生する項目の分類

### 2.1 カテゴリA: テキスト照合が必要な項目（あいまい一致）

名称の表記揺れにより、同一のものが異なるテキストで記録されるケース。

| 項目 | DBテーブル.カラム | 表記揺れパターン |
| --- | --- | --- |
| 契約先名 | companies.name | 下記参照 |
| 現場名 | sites.name + site_sub_items.name | 下記参照 |

#### 表記揺れパターン（契約先名・現場名共通）

| パターン | 例 |
| --- | --- |
| 法人格の省略 | 「株式会社〇〇」vs「(株)〇〇」vs「〇〇」 |
| スペースの違い | 全角スペース vs 半角スペース vs スペースなし |
| カナ表記の揺れ | 全角カナ vs 半角カナ（「エンジニアリング」vs「ｴﾝｼﾞﾆｱﾘﾝｸﾞ」） |
| 略称の使用 | 「西日本高速道路エンジニアリング四国」vs「NEXCO西四国」 |

#### 現場名固有の揺れ

| パターン | 例 |
| --- | --- |
| 階層の結合差異 | Excel「24-1234 〇〇橋補修 点検」= DB「24-1234」+「〇〇橋補修工事」+「点検作業」 |
| 前後スペース・記号 | 「〇〇ビル　巡回警備」vs「〇〇ビル 巡回」 |
| 別名・略称 | 「国道〇号」vs「R〇号」、「県道」vs「府道」 |

### 2.2 カテゴリB: 値の比較が必要な項目

| 項目 | DBテーブル.カラム | 型 |
| --- | --- | --- |
| 区分 | sites.category_id → site_categories | 選択肢 |
| 昼夜 | daily_assignments.shift_type | day/night |
| 人数 | daily_assignments.required_count | 数値 |
| 開始時間 | daily_assignments.start_time | TIME |
| 終了時間 | daily_assignments.end_time | TIME |
| 集合時間 | daily_assignments.meeting_time | TIME |
| 連絡 | daily_assignments.contact | テキスト |

### 2.3 カテゴリC: リスト・テキストの比較

| 項目 | DBテーブル.カラム | 型 |
| --- | --- | --- |
| 配置社員 | assignment_details.employee_id | 社員リスト |
| 備考 | daily_assignments.notes | テキスト |

## 3. 差分処理フロー

### 3.1 全体フロー

```
[ボタン押下]
    ↓
[データ読み込み]
  Excel側データ取得 / DB側データ取得
    ↓
[照合処理]
  ├─ 契約先名・現場名のあいまい一致判定
  ├─ 行のマッチング（Excel行 ↔ DB行）
  └─ 項目ごとの値比較
    ↓
[差分分類]
  ├─ 一致: 処理不要
  ├─ あいまい一致（自動マッチ候補）: 確認必要
  ├─ 値の差分: 選択必要
  ├─ Excel側のみ（新規）: 登録確認必要
  └─ DB側のみ: 削除/保持の判断必要
    ↓
[差分モーダル表示]
  変更がある行のみ表示（一致行は「〇件一致」とサマリ表示）
    ↓
[ユーザー操作]
  項目ごとに反映内容を選択・編集
    ↓
[反映実行]
  選択結果に基づきDB/Excelを更新
```

### 3.2 行マッチングの流れ

```
[Excel行]
    ↓
[完全一致チェック]
  契約先名 + 現場名 + 昼夜 が完全一致するDB行があるか？
  → YES: マッチ確定 → 項目ごとの値比較へ
  → NO: ↓
    ↓
[あいまい一致チェック]
  正規化して比較（後述のアルゴリズム）
  → 候補あり: 「同一と思われます」と表示し確認を求める
  → 候補なし: ↓
    ↓
[新規行として分類]
  「Excelのみ」（取得時）or「DBのみ」（転記時）
```

## 4. あいまい一致アルゴリズム

### 4.1 正規化処理（前処理）

比較の前に両方のテキストに対して以下の正規化を行う。

| 処理 | 内容 | 例 |
| --- | --- | --- |
| 1. 全角→半角変換 | カナ・英数字・記号を半角に統一 | 「ＡＢＣ」→「ABC」 |
| 2. 半角カナ→全角カナ | 半角カナを全角に統一 | 「ｴﾝｼﾞﾆｱﾘﾝｸﾞ」→「エンジニアリング」 |
| 3. スペース正規化 | 全角/半角スペースを半角1つに統一、前後トリム | 「〇〇　ビル」→「〇〇 ビル」 |
| 4. 法人格の除去 | 株式会社、(株)、㈱、有限会社、(有) 等を除去 | 「株式会社〇〇」→「〇〇」 |
| 5. 括弧の統一 | 全角括弧を半角に統一 | 「（四国）」→「(四国)」 |

### 4.2 類似度判定

正規化後のテキストに対して類似度を計算する。

| 判定 | 類似度 | 処理 |
| --- | --- | --- |
| 完全一致 | 100% | 自動マッチ（差分なし） |
| 高類似度 | 80%以上 | 自動マッチ候補として提示し、ユーザー確認 |
| 中類似度 | 50%以上 | 候補リストとして表示（ユーザーが選択） |
| 低類似度 | 50%未満 | 新規として扱う |

#### 類似度計算方法

- **レーベンシュタイン距離ベース**: 正規化後の文字列間の編集距離から類似度を算出
- **部分一致ボーナス**: 一方が他方を含む場合（「〇〇ビル」⊂「〇〇ビル 巡回警備」）にボーナス加算
- **トークン一致**: スペース区切りの各トークンの一致率も考慮

### 4.3 現場名の階層結合照合

Excelの1セルに入った現場名を、DBの階層構造（sites + site_sub_items）と照合する。

```
Excel: "24-1234 〇〇橋補修 点検"
    ↓ スペースでトークン分割
    ["24-1234", "〇〇橋補修", "点検"]
    ↓ DB階層と照合
    sites.name = "24-1234" (部分一致)
    site_sub_items[0].name = "〇〇橋補修工事" (部分一致 "〇〇橋補修" ⊂ "〇〇橋補修工事")
    site_sub_items[1].name = "点検作業" (部分一致 "点検" ⊂ "点検作業")
    → 高類似度でマッチ候補
```

### 5.2 差分行の種類と表示

| 種類 | ラベル | 色 | 内容 |
| --- | --- | --- | --- |
| あいまい一致 | 【あいまい一致】 | 黄色系 | 表記揺れで同一と推定。表記選択＋項目差分の処理 |

| DBのみ | 【DBのみ】 | 赤色系 | DBにあり Excelにない → 保持 or 削除を選択 |

### 5.3 項目ごとの操作

各項目で以下の操作が可能:

| 操作 | 説明 |
| --- | --- |
| Excel側を採用（●） | Excel側の値をラジオボタンで選択 |
| DB側を採用（●） | DB側の値をラジオボタンで選択 |
| 編集 | 選択した値をその場で編集して別の値に変更 |

### 5.4 一括操作

| ボタン | 動作 |
| --- | --- |
| すべてExcel側 | 全差分項目をExcel側に一括選択 |
| すべてDB側 | 全差分項目をDB側に一括選択 |
| 反映実行 | 選択結果をDB/Excelに反映 |

### 5.5 サマリバー

モーダル上部にサマリ情報を表示:
- **一致**: 差分なしの行数
- **差分あり**: 値が異なる行数
- **新規**: 片方にしかない行数（Excelのみ / DBのみ）

## 6. 処理詳細

### 6.1 受注簿データ取得（Excel → DB）

```
1. React Webアプリ上でExcelファイルをアップロード
2. サーバーサイドでExcelファイルをパースしデータを抽出
3. DB側の当日データを取得
4. 行マッチング（契約先名+現場名+昼夜で照合）
5. マッチした行: 項目ごとの値比較
6. マッチしない行: 新規 or DBのみに分類
7. React差分モーダルを表示
8. ユーザーの選択に基づきDBを更新
   - 「Excelのみ」で「登録する」:
     - 契約先が新規なら companies に登録
     - 現場が新規なら sites + site_sub_items に登録
     - daily_assignments を作成
   - 値の差分: 選択された値で daily_assignments 等を更新
```

### 6.2 受注簿へ転記（DB → Excel）

```
1. DB側の当日データを取得
2. React Webアプリ上でExcelファイルをアップロード（既存の受注簿）
3. サーバーサイドでExcelファイルをパースし既存データを抽出
4. 行マッチング（同上）
5. マッチした行: 項目ごとの値比較
6. マッチしない行: 新規 or Excelのみに分類
7. React差分モーダルを表示（方向ラベルが逆）
8. ユーザーの選択に基づきExcelファイルを更新・ダウンロード
```

### 6.3 あいまい一致で「同一」と承認された場合

- ユーザーが表記を選択（Excel側 or DB側）
- 選択された表記でDBを更新（DB側の表記を変更する場合）
- 以降のマッチングではこの対応関係を記憶し、次回から自動マッチ

### 6.4 新規現場の登録フロー

```
1. 「DBに登録する」にチェック
2. 反映実行時に以下を自動作成:
   a. companies: 契約先が未登録なら新規作成
   b. sites: 現場マスタに新規作成
   c. site_sub_items: 階層がある場合は子項目も作成
   d. daily_assignments: 日毎配置を作成
3. 登録後、通常の行として管理
```

## 7. エッジケース

| ケース | 対処 |
| --- | --- |
| 同じ契約先+現場が昼夜両方にある | shift_type（昼/夜）も含めてマッチングする |
| Excelに同一行が重複している | 重複を検知して警告表示 |
| あいまい一致の候補が複数ある | 候補をリスト表示してユーザーに選ばせる |
| Excel側の契約先名が空 | 警告を出し、スキップまたは手動で契約先を指定 |
| DB側のデータが更新中（楽観ロック） | updated_at を比較し、競合があれば警告 |
