# データベース設計書

## 1. 概要

### 1.1 データベース

- **DBMS**: PostgreSQL（Supabase）
- **文字コード**: UTF-8

### 1.2 設計方針

- 正規化を基本としつつ、パフォーマンスを考慮
- 拡張性を確保（将来の機能追加に対応）
- 履歴管理のためのタイムスタンプを全テーブルに設置
- フロントエンド非依存: React Webアプリ（先行開発）およびExcel VBA UI（後続開発）の両方からSupabase REST APIで共通利用

## 3. テーブル定義

### 3.0 group_companies（グループ会社マスタ）

自社グループ会社（東央警備、Nikkeiホールディングス等）を管理するマスタ。
取引先（元請け先）とは異なり、自社側の所属会社を表す。
各現場（sites）はいずれかのグループ会社に紐付く。

| カラム名         | 型           | NULL | 説明                           |
| ---------------- | ------------ | ---- | ------------------------------ |
| id               | UUID         | NO   | 主キー                         |
| code             | VARCHAR(20)  | NO   | グループ会社コード（ユニーク） |
| name             | VARCHAR(100) | NO   | グループ会社名                 |
| badge_color      | VARCHAR(7)   | YES  | ボーダー色（例: #93c5fd）      |
| badge_text_color | VARCHAR(7)   | YES  | バッジ文字色（例: #ffffff）    |
| sort_order       | INTEGER      | YES  | 表示順                         |
| is_active        | BOOLEAN      | NO   | 有効フラグ（デフォルト: true） |
| created_at       | TIMESTAMPTZ  | NO   | 作成日時                       |
| updated_at       | TIMESTAMPTZ  | NO   | 更新日時                       |

```sql
CREATE TABLE group_companies (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    code VARCHAR(20) NOT NULL UNIQUE,
    name VARCHAR(100) NOT NULL,
    badge_color VARCHAR(7) DEFAULT '#718096',
    badge_text_color VARCHAR(7) DEFAULT '#ffffff',
    sort_order INTEGER DEFAULT 0,
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 初期データ
INSERT INTO group_companies (code, name, badge_color, badge_text_color, sort_order) VALUES
    ('TOUO', '東央警備', '#93c5fd', '#ffffff', 1),
    ('NIKKEI', 'Nikkeiホールディングス', '#fca5a5', '#ffffff', 2);
```

### 3.2.1 category_badges（区分バッジマスタ）

区分（site_categories）に紐づく子バッジの定義。
セル編集モーダルで区分に応じた子バッジをチップ選択できる。
親バッジ＝区分名（読み取り専用）、子バッジ＝本テーブルで定義された選択肢。

| カラム名        | 型           | NULL | 説明                                              |
| --------------- | ------------ | ---- | ------------------------------------------------- |
| id              | UUID         | NO   | 主キー                                            |
| category_id     | UUID         | NO   | 区分ID（FK）                                      |
| parent_badge_id | UUID         | YES  | 親バッジID（自己参照FK、NULLなら子バッジ）        |
| level           | INTEGER      | NO   | 階層レベル（1=子バッジ, 2=孫バッジ）              |
| code            | VARCHAR(20)  | NO   | バッジコード                                      |
| name            | VARCHAR(50)  | NO   | バッジ名（表示用）                                |
| color           | VARCHAR(7)   | YES  | バッジ背景色                                      |
| text_color      | VARCHAR(7)   | YES  | バッジ文字色                                      |
| sort_order      | INTEGER      | YES  | 表示順                                            |
| is_active       | BOOLEAN      | NO   | 有効フラグ                                        |
| created_at      | TIMESTAMPTZ  | NO   | 作成日時                                          |
| updated_at      | TIMESTAMPTZ  | NO   | 更新日時                                          |

※ 親バッジは `site_categories.name` をそのまま使用（読み取り専用表示）
※ 子バッジ: `parent_badge_id = NULL`, `level = 1`（区分直下）
※ 孫バッジ: `parent_badge_id = 子バッジID`, `level = 2`（子バッジの詳細項目）
※ 子バッジが0件の区分もある（施設、イベント等）。その場合、UIでは「バッジなし」と表示し、追加ボタンから新規作成可能
※ ユーザーがモーダルから新規バッジを追加した場合、本テーブルにINSERTされる

```sql
CREATE TABLE category_badges (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    category_id UUID NOT NULL REFERENCES site_categories(id) ON DELETE CASCADE,
    parent_badge_id UUID REFERENCES category_badges(id) ON DELETE CASCADE,
    level INTEGER NOT NULL DEFAULT 1 CHECK (level IN (1, 2)),
    code VARCHAR(20) NOT NULL,
    name VARCHAR(50) NOT NULL,
    color VARCHAR(7) DEFAULT '#718096',
    text_color VARCHAR(7) DEFAULT '#ffffff',
    sort_order INTEGER DEFAULT 0,
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(category_id, code)
);

CREATE INDEX idx_category_badges_category ON category_badges(category_id);
CREATE INDEX idx_category_badges_parent ON category_badges(parent_badge_id);

-- 初期データ（高速系）
INSERT INTO category_badges (category_id, code, name, sort_order)
SELECT sc.id, b.code, b.name, b.sort_order
FROM site_categories sc
CROSS JOIN (VALUES
    ('LANE_REGULATION', '車線規制', 1),
    ('SHOULDER_REGULATION', '路肩規制', 2),
    ('BOOTH_REGULATION', 'ブース規制', 3),
    ('SECURITY_GUARD', '保安員', 4)
) AS b(code, name, sort_order)
WHERE sc.code = 'HIGHWAY';

-- 初期データ（交通系）
INSERT INTO category_badges (category_id, code, name, sort_order)
SELECT sc.id, b.code, b.name, b.sort_order
FROM site_categories sc
CROSS JOIN (VALUES
    ('ALTERNATE_TRAFFIC', '片側交互', 1),
    ('ROAD_CLOSURE', '通行止め', 2)
) AS b(code, name, sort_order)
WHERE sc.code = 'TRAFFIC';
```

### 3.2.2 site_sub_items（現場子項目）

現場の階層構造を管理するテーブル。PJNo.→工事名→作業内容のような階層を表現。

| カラム名   | 型           | NULL | 説明                                    |
| ---------- | ------------ | ---- | --------------------------------------- |
| id         | UUID         | NO   | 主キー                                  |
| site_id    | UUID         | NO   | 現場ID（FK）                            |
| parent_id  | UUID         | YES  | 親項目ID（FK、NULLなら第1階層子）       |
| label      | VARCHAR(50)  | NO   | 項目ラベル（例: 工事名、作業内容）       |
| name       | VARCHAR(100) | NO   | 項目名（値）                            |
| level      | INTEGER      | NO   | 階層レベル（1=子、2=孫）                |
| sort_order | INTEGER      | YES  | 表示順                                  |
| is_active  | BOOLEAN      | NO   | 有効フラグ                              |
| created_at | TIMESTAMPTZ  | NO   | 作成日時                                |
| updated_at | TIMESTAMPTZ  | NO   | 更新日時                                |

※ `label`: 現場ごとにカスタマイズ可能な項目種別名。高速系は「工事名」「作業内容」、施設系は「業務内容」など、区分や現場に応じて自由に設定できる。

```sql
CREATE TABLE site_sub_items (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    site_id UUID NOT NULL REFERENCES sites(id) ON DELETE CASCADE,
    parent_id UUID REFERENCES site_sub_items(id) ON DELETE CASCADE,
    label VARCHAR(50) NOT NULL DEFAULT '項目',
    name VARCHAR(100) NOT NULL,
    level INTEGER NOT NULL CHECK (level BETWEEN 1 AND 2),
    sort_order INTEGER DEFAULT 0,
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_site_sub_items_site ON site_sub_items(site_id);
CREATE INDEX idx_site_sub_items_parent ON site_sub_items(parent_id);
```

**使用例**:

西日本高速道路エンジニアリング四国㈱の場合:
```
sites: name = "24-1234" (hierarchy_depth = 3)
  └─ site_sub_items: label = "工事名", name = "〇〇橋補修工事" (level = 1)
       └─ site_sub_items: label = "作業内容", name = "点検作業" (level = 2, parent_id = 上の項目)

→ 計画書業務名: "24-1234 > 〇〇橋補修工事 > 点検作業"
```

Nikkei施設系の場合:
```
sites: name = "〇〇ビル" (hierarchy_depth = 2)
  └─ site_sub_items: label = "業務内容", name = "巡回警備" (level = 1)

→ 計画書業務名: "〇〇ビル > 巡回警備"
```

個別業務の場合:
```
sites: name = NULL (hierarchy_depth = 2)
  └─ site_sub_items: label = "作業内容", name = "交通誘導" (level = 1)

→ 計画書業務名: "交通誘導"（親業務名なし）
```

### 3.3.1 employee_conflicts（社員配置制約）

人間関係（相性）により、同じ現場に配置すべきでない社員同士を管理。
配置時に注意喚起アイコンを表示する。

| カラム名      | 型          | NULL | 説明       |
| ------------- | ----------- | ---- | ---------- |
| id            | UUID        | NO   | 主キー     |
| employee_id_1 | UUID        | NO   | 社員1（FK）|
| employee_id_2 | UUID        | NO   | 社員2（FK）|
| is_active     | BOOLEAN     | NO   | 有効フラグ |
| created_at    | TIMESTAMPTZ | NO   | 作成日時   |
| updated_at    | TIMESTAMPTZ | NO   | 更新日時   |

```sql
CREATE TABLE employee_conflicts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    employee_id_1 UUID NOT NULL REFERENCES employees(id) ON DELETE CASCADE,
    employee_id_2 UUID NOT NULL REFERENCES employees(id) ON DELETE CASCADE,
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    -- 重複防止: employee_id_1 < employee_id_2 の制約
    CONSTRAINT chk_employee_order CHECK (employee_id_1 < employee_id_2),
    -- 同じ組み合わせは1つのみ
    UNIQUE(employee_id_1, employee_id_2)
);

CREATE INDEX idx_employee_conflicts_1 ON employee_conflicts(employee_id_1);
CREATE INDEX idx_employee_conflicts_2 ON employee_conflicts(employee_id_2);
```

**UI動作**:
- 同じ現場・同じ日・同じシフトに制約関係の社員が配置された場合
- 該当社員の横に警告アイコンを表示
- ホバーで「〇〇さんと配置制約があります」等のツールチップ

**制約チェックSQL例**:
```sql
-- 特定の配置で制約関係の社員がいるかチェック
SELECT ec.*, e1.name as employee1_name, e2.name as employee2_name
FROM employee_conflicts ec
JOIN employees e1 ON ec.employee_id_1 = e1.id
JOIN employees e2 ON ec.employee_id_2 = e2.id
WHERE ec.is_active = true
  AND (ec.employee_id_1 IN (:assigned_employee_ids)
       OR ec.employee_id_2 IN (:assigned_employee_ids))
  AND ec.employee_id_1 IN (:assigned_employee_ids)
  AND ec.employee_id_2 IN (:assigned_employee_ids);
```

### 3.6 daily_site_orders（日毎受注）

営業担当者が契約先から受けた受注情報を管理するテーブル。
「どの現場に、いつ、何名必要か」という受注の基本情報を保持する。

**業務背景**: 営業担当者が契約先から連絡を受け、システムに入力する。営業担当者が直接入力できない場合は、
配置担当者に連絡して代理入力してもらう。複数の営業担当者が同じ受注情報を変更する可能性があるため、

| start_time      | TIME         | YES  | 開始時間                                                    |
| end_time        | TIME         | YES  | 終了時間                                                    |
| contact         | VARCHAR(100) | YES  | 連絡                                                        |
| supervisor_name | VARCHAR(100) | YES  | 現場監督名                                                  |

| notes           | TEXT         | YES  | 備考                                                        |
| badge_child_ids | UUID[]       | YES  | 選択済み子バッジID（category_badges.id の配列）             |

### 3.7.2 order_change_history（受注変更履歴）

日毎受注（daily_site_orders）の全変更を記録するテーブル。変更されたカラム名・変更前の値・変更後の値を記録し、
任意の過去バージョンへのロールバックを可能にする。

**業務背景**: 複数の営業担当者が受注情報を入力・変更し、配置担当者が代理入力やキャンセル処理を行う運用において、
聞き間違いによる誤キャンセル、誤った人数変更、他の営業担当者による誤情報の上書きが発生する。
変更履歴により「誰がいつ何を変更したか」を追跡し、誤操作時のロールバックを可能にする。

| カラム名       | 型           | NULL | 説明                                                |
| -------------- | ------------ | ---- | --------------------------------------------------- |
| id             | UUID         | NO   | 主キー                                              |
| order_id       | UUID         | NO   | 日毎受注ID（FK）                                    |
| version        | INTEGER      | NO   | この変更後のバージョン番号                          |
| operation_type | VARCHAR(20)  | NO   | 操作種別（create/update/cancel/restore/rollback/merge） |
| changed_fields | JSONB        | NO   | 変更内容 [{field, old_value, new_value}, ...]       |
| change_reason  | TEXT         | YES  | 変更理由（任意入力）                                |
| changed_by     | UUID         | NO   | 変更者（FK → employees）                            |
| changed_at     | TIMESTAMPTZ  | NO   | 変更日時                                            |

**operation_type の値**:

| 値       | 説明                                               |
| -------- | -------------------------------------------------- |
| create   | 新規受注の入力                                     |
| update   | 通常の更新（人数変更、時間変更等）                 |
| cancel   | その日のキャンセル処理（status → cancelled）       |
| restore  | キャンセルからの復元（status → active）            |
| rollback | 過去バージョンへのロールバック                     |
| merge    | 競合解決時のスマートマージ（非重複項目の統合）     |

**changed_fields の形式**:

```json
[
  { "field": "required_count", "old_value": "3", "new_value": "2" },
  { "field": "start_time", "old_value": "08:00", "new_value": "09:00" },
  { "field": "status", "old_value": "active", "new_value": "cancelled" }
]
```

```sql
CREATE TABLE order_change_history (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id UUID NOT NULL REFERENCES daily_site_orders(id) ON DELETE CASCADE,
    version INTEGER NOT NULL,
    operation_type VARCHAR(20) NOT NULL CHECK (operation_type IN ('create', 'update', 'cancel', 'restore', 'rollback', 'merge')),
    changed_fields JSONB NOT NULL DEFAULT '[]'::jsonb,
    change_reason TEXT,
    changed_by UUID NOT NULL REFERENCES employees(id),
    changed_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(order_id, version)
);

CREATE INDEX idx_order_change_history_order ON order_change_history(order_id);
CREATE INDEX idx_order_change_history_changed_by ON order_change_history(changed_by);
CREATE INDEX idx_order_change_history_changed_at ON order_change_history(changed_at);
```

**楽観的ロック（競合検知）の動作**:

1. 画面表示時に受注の `version` を取得
2. 保存時にDBの `version` と照合
3. 一致すれば保存成功 → `version` をインクリメント、`order_change_history` に記録
4. 不一致なら他者が先に変更済み → **スマートマージ判定**を実行（下記参照）

**スマートマージ（競合解決）の動作**:

バージョン不一致を検知した場合、自分の変更項目と他者の変更項目を比較し、競合の種類に応じた解決方法を提示する。

**ケースA: 変更項目が重複しない場合（自動マージ候補）**
- 例: 営業Aが `start_time` を変更、営業Bが `required_count` を変更
- ダイアログで両者の変更内容を表示し、以下の選択肢を提示:
  - **両方の変更をマージ**: 営業Aの変更と営業Bの変更を統合して保存（`operation_type = 'merge'`）
  - **自分の変更のみ上書き**: 他者の変更を破棄し、自分の変更で上書き
  - **キャンセル**: 保存を中止し、最新データを再取得

**ケースB: 変更項目が重複する場合（手動選択必須）**
- 例: 営業Aも営業Bも `required_count` を変更
- 競合ダイアログで重複項目ごとに以下を表示:
  - 変更前の値（ベースバージョン）
  - 他者の変更後の値（現在のDB値）
  - 自分の変更後の値
- 各項目について、他者の値と自分の値のどちらを採用するか選択
- 非重複項目がある場合は、それらも含めてマージ
- 最終的な変更内容を確認後に保存（`operation_type = 'merge'`）

**マージ後の履歴記録**:
- マージ操作は `operation_type = 'merge'` として `order_change_history` に記録
- `changed_fields` には最終的に適用された変更の差分を記録
- `change_reason` に「バージョンX（営業Aの変更）とマージ」等の自動メモを付与

**ロールバックの動作**:

1. 変更履歴一覧からロールバック先のバージョンを選択
2. 選択バージョンの `changed_fields` から、そのバージョン時点の値を復元
3. ロールバック自体も `operation_type = 'rollback'` として履歴に記録
4. 営業担当者・配置担当者・管理者がロールバック可能

### 3.9 連続勤務（自動検知）

連続勤務はテーブルで管理せず、**配置データから自動検知**する機能として実装。

#### 検知ロジック

**検知範囲**: 同日内 + 日をまたぐパターンの両方

| パターン   | 説明                     | 検知条件                               |
| ---------- | ------------------------ | -------------------------------------- |
| 昼→夜     | 同日の昼勤務後に夜勤務   | 同一社員が同日に昼・夜両方に配置       |
| 夜→昼     | 前日夜勤務後に翌日昼勤務 | 同一社員が前日夜に配置＆当日昼に配置   |
| 昼→夜→昼 | 昼から夜を経て翌日昼     | 同一社員が昼→夜→翌日昼に連続配置     |
| 夜→昼→夜 | 夜から昼を経て同日夜     | 同一社員が前日夜→当日昼→当日夜に配置 |

#### 表示方法

- **アイコン・バッジ**: 連続勤務パターンを示すアイコンを社員名の横に表示
- **背景色**: 連続勤務中の社員を背景色で視覚的に強調

#### 検知SQL例

```sql
-- 特定日の社員の連続勤務パターンを検知
WITH employee_shifts AS (
    SELECT
        ad.employee_id,
        da.assignment_date,
        da.shift_type,
        da.id as assignment_id
    FROM assignment_details ad
    JOIN daily_assignments da ON ad.daily_assignment_id = da.id
    WHERE da.assignment_date BETWEEN :target_date - 1 AND :target_date
),
patterns AS (
    SELECT
        e1.employee_id,
        -- 同日の昼→夜
        CASE WHEN EXISTS (
            SELECT 1 FROM employee_shifts e2
            WHERE e2.employee_id = e1.employee_id
            AND e2.assignment_date = e1.assignment_date
            AND e1.shift_type = 'day' AND e2.shift_type = 'night'
        ) THEN true ELSE false END as day_to_night,
        -- 前日夜→当日昼
        CASE WHEN EXISTS (
            SELECT 1 FROM employee_shifts e2
            WHERE e2.employee_id = e1.employee_id
            AND e2.assignment_date = e1.assignment_date - 1
            AND e2.shift_type = 'night' AND e1.shift_type = 'day'
        ) THEN true ELSE false END as night_to_day
    FROM employee_shifts e1
    WHERE e1.assignment_date = :target_date
)
SELECT
    employee_id,
    CASE
        WHEN day_to_night AND night_to_day THEN 'night_day_night'
        WHEN day_to_night THEN 'day_to_night'
        WHEN night_to_day THEN 'night_to_day'
        ELSE null
    END as continuous_pattern
FROM patterns
WHERE day_to_night OR night_to_day;
```

## 5. RLS（Row Level Security）ポリシー

Supabaseで認証されたユーザーのみアクセス可能にする設定。

```sql
-- 全テーブルにRLSを有効化
ALTER TABLE companies ENABLE ROW LEVEL SECURITY;
ALTER TABLE sites ENABLE ROW LEVEL SECURITY;
ALTER TABLE employees ENABLE ROW LEVEL SECURITY;
-- ... 他のテーブルも同様

-- 認証済みユーザーは全データにアクセス可能（社内システムのため）
CREATE POLICY "Authenticated users can read all data" ON companies
    FOR SELECT USING (auth.role() = 'authenticated');

CREATE POLICY "Authenticated users can insert data" ON companies
    FOR INSERT WITH CHECK (auth.role() = 'authenticated');

CREATE POLICY "Authenticated users can update data" ON companies
    FOR UPDATE USING (auth.role() = 'authenticated');

-- ... 他のテーブルも同様
```
